# Onboarding Principles

DroidPuppy should not stop at the first gimmick.

A cool first demo gets attention.
A guided second and third step create retention.

## Principle 1: show the next move

After a user successfully does something for the first time, DroidPuppy should point to the natural next step.

Examples:
- After opening Brave, suggest reading a page.
- After reading a page, suggest listing links or taking a screenshot.
- After checking reconnect state, suggest quick reconnect or full repair.

## Principle 2: plain language before jargon

Users should be able to think in terms like:
- open wifi
- test the browser
- reconnect the phone
- capture the screen
- share the repo

The technical layer should exist underneath, but not be required to start.

## Principle 3: diagnostics should become setup help

If a capability is missing, the tool should not dead-end.
It should explain what is missing and what the easiest next step is.

## Principle 4: wizard-like flows beat documentation walls

People on phones need short, guided steps.

Not:
- giant setup manuals
- unexplained command dumps

But:
- do this first
- now do this
- here is why it matters

## Principle 5: power should unfold gradually

DroidPuppy should reveal depth in layers:
1. simple commands
2. friendly workflows
3. guided setup
4. deeper debugging and automation

That is how serious capability becomes usable capability.
